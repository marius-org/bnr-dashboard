What is Cloud-Init?
☁️ Cloud-Init

First-boot VM configuration — automatic, zero interaction

Cloud-init is the industry standard for bootstrapping virtual machines. The moment a VM starts for the first time, cloud-init reads a configuration source and sets everything up before you ever touch the keyboard — SSH keys, hostname, static IP, users, packages.

This is the same technology powering every major cloud provider. When you launch an EC2 instance on AWS and it already has your SSH key ready — that's cloud-init. Proxmox implements the exact same standard via a virtual CD-ROM drive (the --ide2 cloudinit drive) that the VM reads on first boot.

Without cloud-init: boot each VM → open console → manually set IP → add SSH key → set hostname → create user. Repeated 5 times.

With cloud-init: clone template → VM boots → 30 seconds later it's fully configured and reachable.

What cloud-init injected into each VM
SSH Public Key

Your Windows workstation key + control-node key added to authorized_keys. Login without password from day one.

Hostname

Each clone gets its own name: k3s-master-01, k3s-worker-02, etc. Set automatically at first boot.

Static IP Address

192.168.1.51 through .55 assigned per VM. No DHCP, no guessing which IP a VM got.

Default User

ubuntu user created with passwordless sudo. Ready for Ansible to connect and configure.

Gateway & DNS

192.168.1.1 as gateway, 8.8.8.8 as DNS. VM has full internet access on first boot.

Template Creation — Every Command Explained

These commands were run directly on the Proxmox host (pve12) via SSH. Together they download Ubuntu, customize the disk image, build VM 9000, and lock it as a reusable template. This is your private AMI.

01Download the Ubuntu Cloud Image
wget https://cloud-images.ubuntu.com/noble/current/noble-server-cloudimg-amd64.img -O /tmp/ubuntu-24.04-cloud.img
Downloads the official Ubuntu 24.04 LTS cloud image from Canonical. This is not a regular ISO — it's a pre-built, compressed disk image specifically designed for virtual environments. It's minimal (~600MB), boots in seconds, and has cloud-init already baked in. The -O flag saves it with a clean filename to /tmp.
02Install libguestfs-tools
apt install -y libguestfs-tools
Installs the libguestfs toolkit — a suite of tools for reading and modifying VM disk images without ever booting them. Think of it as mounting a hard drive, chrooting inside, making changes, then unmounting. The -y flag skips the confirmation prompt. Required for the next step.
03Inject qemu-guest-agent into the disk image
virt-customize -a /tmp/ubuntu-24.04-cloud.img --install qemu-guest-agent
virt-customize opens the disk image and runs apt install qemu-guest-agent inside it — without booting the VM. The guest agent is a small daemon that runs inside the VM and lets Proxmox communicate with it: clean shutdown, IP address detection, filesystem freeze for live backups. Without it, Proxmox is blind to what's inside the VM.
04Create an empty VM shell in Proxmox
qm create 9000 --name "ubuntu-2404-template" --memory 2048 --cores 2 --net0 virtio,bridge=vmbr0
Creates VM ID 9000 as an empty hardware definition — no disk yet. virtio is the paravirtualized network driver (much faster than emulated Intel e1000). bridge=vmbr0 connects it to the physical network bridge so VMs can reach your LAN and internet. Memory and cores are just defaults for the template; clones can override these.
05Import the disk image into Proxmox storage
qm importdisk 9000 /tmp/ubuntu-24.04-cloud.img local-lvm
Takes the downloaded .img file and imports it into Proxmox's LVM storage pool, associating it with VM 9000. After this, the disk exists in storage as vm-9000-disk-0 but is not yet connected to the VM — it's an unattached volume.
06Attach the disk to the VM
qm set 9000 --scsihw virtio-scsi-pci --scsi0 local-lvm:vm-9000-disk-0
Connects the imported disk to VM 9000 as the primary drive on scsi0. virtio-scsi-pci is the paravirtualized SCSI controller — significantly faster than emulated IDE or SATA because it avoids hardware emulation overhead. Now the VM has a bootable operating system disk.
07Add the cloud-init drive
qm set 9000 --ide2 local-lvm:cloudinit
This is the magic step. Adds a virtual CD-ROM drive on IDE2 that Proxmox uses to pass cloud-init configuration into the VM at first boot. When you clone this template and set SSH keys, IP, hostname in Proxmox — it writes them to this drive. The VM reads it on boot and applies everything automatically. No this drive = no cloud-init injection.
08Set boot order
qm set 9000 --boot c --bootdisk scsi0
Tells the VM to boot from disk (c = disk in BIOS notation) and specifically from scsi0. Without this the VM's BIOS might try to PXE boot over the network or look at other drives first, causing a boot failure.
09Configure serial console
qm set 9000 --serial0 socket --vga serial0
Adds a serial port and redirects display output to it. Cloud images often send console output to the serial port instead of VGA. This lets you access the VM via Proxmox's built-in terminal (noVNC) even when the network isn't configured yet — critical for debugging first-boot issues or misconfigured IPs.
10Enable guest agent integration
qm set 9000 --agent enabled=1
Tells Proxmox to actively communicate with the qemu-guest-agent daemon installed in step 03. Enables: clean graceful shutdown (vs hard power off), live IP address reporting in the Proxmox UI, filesystem freeze/thaw for consistent live backups, and guest OS information display.
11Convert VM to template
qm template 9000
Locks VM 9000 permanently — it can no longer be started directly. It now exists only as a source for cloning. When Proxmox clones a template on LVM storage, it uses linked clones — the clone shares the base disk blocks and only stores its own changes (copy-on-write). This means cloning takes seconds and uses minimal extra storage. Your private AMI is ready.

🖱️ Manual Clone (control-node)
Done once via Proxmox web UI. Right-click template → Clone → set VM ID 150, name control-node, full clone. Then set cloud-init values in the VM's Cloud-Init tab: user=ubuntu, SSH key, IP=192.168.1.50/24, gateway=192.168.1.1. Start the VM. This is the chicken-and-egg exception — control-node runs Terraform, so it can't provision itself.
🏗️ Terraform Clone (all other VMs)
Terraform calls the Proxmox API using the bpg/proxmox provider. For each entry in the locals map (151–155), it clones template 9000, sets cloud-init parameters (IP, hostname, SSH key), and starts the VM. All 5 VMs are provisioned in parallel in under 2 minutes. Fully repeatable — terraform destroy then terraform apply rebuilds everything from scratch.
Kubernetes / k3s — For Complete Beginners
🏭

Think of it as a smart factory manager

You don't say "go to machine #3 and start the conveyor belt". You say "I need 2 units of product X running at all times". The factory manager (Kubernetes) figures out which machines are free, places the work, monitors it, and if a machine breaks — automatically moves the work to another one. You only describe the desired outcome, never the manual steps.

🧠 Control Plane (Masters × 3)
The brain — makes all decisions
API Server
The single entry point for everything. kubectl, Terraform, GitHub Actions — all talk to this. It's the front door of the cluster.
etcd
A distributed key-value database that stores the entire cluster state. How many pods should exist, what's currently running, all config. The source of truth. Replicated across all 3 masters — this is what provides HA.
Scheduler
Watches for new pods with no assigned node and picks the best worker to run them on — based on available CPU, memory, and other constraints.
Controller Manager
Runs control loops that continuously compare desired state vs actual state. If you said "I want 2 pods" and one crashes, the controller notices and tells the scheduler to create a replacement.
⚙️ Data Plane (Workers × 2)
The muscle — runs your workloads
kubelet
The agent running on every worker node. Receives pod assignments from the control plane and tells the container runtime (containerd) to start or stop containers. Also runs health checks.
containerd
The container runtime — actually pulls Docker images and runs containers. k3s uses containerd directly, no Docker daemon needed on workers.
kube-proxy
Maintains network rules on each node so that Services work correctly — traffic arriving for a Service IP gets routed to the right pod, wherever it's running in the cluster.
Pods (your app)
The smallest deployable unit. A pod wraps one or more containers and gives them a shared IP and storage. Your BNR Dashboard runs as 2 pods — one on each worker node.
The 4 Core Kubernetes Objects — What You Actually Write
📦
Pod
lowest level unit

One running instance of your app container. Has its own IP inside the cluster but it's ephemeral — if it crashes or gets replaced, it gets a new IP. You almost never create pods directly.

Like a single worker at a desk
🔁
Deployment
k8s/deployment.yaml

Wraps pods and adds intelligence: "keep 2 replicas of this image running at all times". If one pod dies, Deployment creates a new one. Handles rolling updates — swaps old pods for new ones one at a time with zero downtime.

Like a manager who ensures 2 workers are always at their desks
🔗
Service
k8s/service.yaml

A stable, permanent IP and DNS name that sits in front of your pods. Pods come and go and change IPs — the Service IP never changes. It load-balances traffic across all healthy pods automatically. Type ClusterIP = internal only.

Like a reception desk that always knows where to route calls
🌐
Ingress
k8s/ingress.yaml

The external door into the cluster. Receives HTTP traffic from outside (via the Nginx Ingress Controller on NodePort 32525) and routes it to the right Service based on path or hostname rules. Without Ingress, your app is unreachable from outside the cluster.

Like the building's front door with a directory board

☸️ Kubernetes (full)

The complete upstream project. Powerful and flexible but complex to install and operate. Requires dedicated tooling (kubeadm, etc.), heavier resource usage. Used in production clouds (GKE, EKS, AKS).

VS
k3s (what we used)

A certified, lightweight Kubernetes distribution by Rancher. Same API, same kubectl, same manifests — but packaged as a single ~70MB binary. Replaces etcd with SQLite by default (we use etcd for HA). Perfect for homelabs, edge, and IoT. Installs in 30 seconds.

CI/CD Pipeline — Step by Step
🚀 What happens every time you run git push origin main
01
Push Detected

GitHub detects a push to main and triggers the workflow in .github/workflows/deploy.yml. The self-hosted control-node runner picks up the job.

02
Code Checkout

The runner clones the repository using actions/checkout@v4. This gives the runner access to the Dockerfile and all app source code.

03
Docker Build and Push

Builds a new Docker image using docker/build-push-action. Two tags pushed to Docker Hub: mariuseu/bnr-dashboard:latest and mariuseu/bnr-dashboard:<git-sha>.

04
k3s Deployment Update

kubectl set image updates the deployment to the new SHA-tagged image. Kubernetes starts a rolling update — new pods start before old ones stop. Zero downtime guaranteed.

05
Rollout Verified

kubectl rollout status waits for all pods to pass readiness probes. If anything fails, the pipeline fails and the previous version continues serving traffic automatically.

Key Concepts Mastered

🔄 Idempotency

Running Terraform or Ansible 10 times gives the same result as running it once. Tools check current state before making changes — safe to re-run anytime.

📝 Declarative vs Imperative

Terraform and Kubernetes are declarative — you describe the desired end state, not the steps. The tool figures out how to make reality match the declaration.

🏗️ Separation of Concerns

Each tool has one job: Proxmox virtualizes, Terraform provisions, Ansible configures, k3s orchestrates, GitHub Actions deploys. Layers are replaceable independently.

🔢 etcd Quorum

With 3 masters, 2 must agree on every write. The cluster tolerates losing 1 master and keeps running. Odd numbers (1,3,5) are always used for control planes.

☁️ Cloud-init

First-boot VM configuration standard. Injects SSH keys, sets hostname, configures IPs, creates users automatically. Same concept as EC2 user-data on AWS.

🔁 Rolling Updates

Kubernetes replaces pods one at a time. New pods must pass readiness probes before old ones terminate. The app stays available throughout the entire update process.

🏷️ Image Tagging with SHA

Every deployment uses the git commit SHA as the image tag. Makes every deployment traceable to exact code and prevents accidental use of stale :latest images.

🩺 Health Probes

Liveness probes restart unhealthy containers. Readiness probes prevent traffic reaching unready pods. Both configured to hit the /health endpoint every 30 seconds.

🔑 Secrets Management

Sensitive values never appear in code. They live in GitHub Secrets or terraform.tfvars (git-ignored). The .gitignore file is the first line of defense against leaks.
